# CryptoScannerX

Асинхронный терминал поиска арбитражных возможностей между криптовалютными биржами (Binance, Bybit, MEXC).

## Возможности

- Получение тикеров через REST API бирж (spot + futures)
- Поиск спредов:
  - Spot ↔ Spot
  - Futures ↔ Futures
  - Spot ↔ Futures (basis)
- **Funding Rate Arbitrage** — поиск пар бирж, где выгодно шортить фьючерс на одной
  и лонговать тот же коин на другой, собирая разницу funding rate (рыночно-нейтрально)
- Расчёт комиссий и чистой ожидаемой прибыли по фиксированному размеру позиции
- Фильтрация по объёму и по списку монет
- **Проверка глубины стакана (slippage)** — для найденных возможностей отдельно
  считается реальная VWAP-цена исполнения на `POSITION_SIZE_USDT`, а не только
  top-of-book bid/ask
- **Проверка идентичности актива (CoinGecko)** — подтверждает, что тикер с
  одинаковым названием на разных биржах — реально один и тот же проект, а не
  разные активы под одним символом
- **История в SQLite** — каждый запуск автоматически дописывает найденные
  возможности в локальную базу для последующего анализа

### Проверка идентичности актива (опционально, но рекомендуется)

Короткие тикеры (например `LA`, `ON`) нередко переиспользуются разными
проектами на разных биржах — без проверки скрипт может принять два разных
актива за один и тот же. Чтобы включить проверку через CoinGecko:

1. Зарегистрируй бесплатный Demo-ключ: https://www.coingecko.com/en/api/pricing
2. Скопируй `.env.example` в `.env` и впиши ключ:
   ```
   COINGECKO_API_KEY=твой_ключ
   ```
   `.env` уже в `.gitignore` — ключ никогда не попадёт в git.

Без ключа скрипт работает как обычно, просто колонка `ID` в таблицах будет
показывать `?` (не проверялось) для всех строк.

**Ограничения проверки:**
- Запросы к CoinGecko троттлятся под лимит бесплатного тарифа
  (`COINGECKO_RATE_LIMIT_PER_MINUTE` в `app/config.py`, по умолчанию 25/мин) —
  при полном скане рынка (без фильтра монет) проверка может занять несколько минут.
- CoinGecko видит в основном **spot**-листинги. Для чисто-фьючерсных пар
  (`FUNDING ARBITRAGE`) и токенизированных акций (`TQQQ`, `NVDL`, `SMCI` и
  подобные) `⚠️`/`?` не обязательно значит "разные активы" — может просто
  значить "не смогли проверить". Для таких строк ручная проверка остаётся
  обязательной.

## Запуск

```bash
# по всему рынку (согласно FILTER_COINS в app/config.py)
python -m app.main

# с фильтром по конкретным монетам
python -m app.main BTC ETH SOL

# с техническими логами (подключение к биржам, количество тикеров на каждом шаге)
python -m app.main BTC ETH SOL -v

# показать последние 20 записей истории вместо нового скана
python -m app.main --history

# показать последние 50 записей
python -m app.main --history 50
```

По умолчанию видны: статус подключения к биржам (✅/❌) и сами таблицы с результатами.
Флаг `-v` / `--verbose` добавляет технические INFO-логи (количество тикеров на
каждом шаге фильтрации, funding-запросы к MEXC и т.д.) — полезно для отладки, если
что-то пошло не так.

## Конфигурация

Основные параметры — в `app/config.py`:

| Параметр | Описание |
|---|---|
| `MIN_SPREAD` | Минимальный сырой спред для поиска (%) |
| `DISPLAY_SPREAD` | Минимальный спред для отображения (%) |
| `MIN_VOLUME_USDT` | Минимальный 24h объём тикера (USDT) |
| `MIN_FUNDING_SPREAD` | Минимальный funding-спред для поиска (%, за период) |
| `DEPTH_LIMIT` | Сколько уровней стакана запрашивать при проверке проскальзывания |
| `POSITION_SIZE_USDT` | Размер позиции для расчёта прибыли (USDT) |
| `FILTER_COINS` | Список монет по умолчанию (`[]` = весь рынок) |

## Проверка глубины стакана (slippage)

Спред между top-of-book bid/ask не гарантирует, что весь объём позиции
(`POSITION_SIZE_USDT`) исполнится по этой цене — если стакан тонкий, реальная
цена исполнения будет хуже. `DepthChecker` запрашивает реальный стакан (несколько
уровней) для уже найденных возможностей (не для всего рынка — под каждый символ
нужен отдельный запрос) и пересчитывает спред как VWAP-цену исполнения.

В таблице `ARBITRAGE` колонки `RAW/FEE/NET/PNL` — по top-of-book цене (как раньше),
`SLIP/REAL NET/REAL PNL` — с учётом реальной глубины. Прочерк (`—`) означает, что
проверка глубины не проводилась или запрос стакана не удался — тогда стоит
доверять только исходным `RAW/NET` цифрам и проверить вручную перед исполнением.

## Funding Rate Arbitrage — как это работает

Funding rate у бирж-перпетуалов выплачивается каждые 8 часов между лонгами и шортами.
Если на одной бирже funding rate высокий положительный, а на другой — низкий или
отрицательный, можно:

1. Шортить фьючерс на бирже с более высоким funding rate (шорты получают funding)
2. Лонговать тот же коин на бирже с более низким/отрицательным funding rate

Цена почти не важна — позиция рыночно-нейтральна, прибыль — это разница funding-выплат
каждый период. Binance и Bybit отдают funding rate бесплатно вместе с обычными
тикерами; у MEXC funding rate можно получить только по одному символу за запрос,
поэтому donабор идёт батчами с паузой между ними, чтобы уважать лимит биржи
(20 запросов/2с). При полном скане рынка (без фильтра монет) это может занять
1-2 минуты — для более быстрых прогонов используй фильтр по конкретным монетам.

## Структура проекта

```
app/
    calculators/     # расчёт комиссий и прибыли
```

## Версионирование

Проект использует [Semantic Versioning](https://semver.org/lang/ru/) (`MAJOR.MINOR.PATCH`):

- **PATCH** (0.6.**1**) — багфиксы без нового функционала
- **MINOR** (0.**7**.0) — новый функционал, обратно совместимый
- **MAJOR** (**1**.0.0) — ломающие изменения или готовность к продакшену

Текущая версия — в [`app/__version__.py`](app/__version__.py), также доступна через:
```bash
python -m app.main --version
```

Каждая версия отмечена git-тегом (`git tag`) и описана в [CHANGELOG.md](CHANGELOG.md).
Список тегов: `git tag -l` или на GitHub во вкладке **Tags**/**Releases**.
